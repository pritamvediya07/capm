# Phase 5 — Operations Guide (connect, run, monitor, recover)

**Audience:** whoever drives the deployed Phase-5 server day-to-day.
**Target box:** ~128 CPU cores · 256+ GB RAM · 96 GB VRAM (single GPU, shared).
**Companions:** [`PHASE5_AGENT_GUIDE.md`](PHASE5_AGENT_GUIDE.md) (start-here for Claude),
[`PHASE5_IMPLEMENTATION.md`](PHASE5_IMPLEMENTATION.md) (architecture/why),
[`PHASE5_BUILD_PLAN.md`](PHASE5_BUILD_PLAN.md) (build spec/files). **This doc is the "how to
operate" runbook.** It documents the *intended* operational interface; commands/endpoints
exist once the package (`capm-p5/`) is built per the build plan.

> Mental model recap: you deploy once on the 96 GB box during your access window
> (`./run.sh up` — which also **trains + freezes one probe per model**), the server stays up
> as a Docker daemon, and thereafter you drive it **remotely over HTTPS+mTLS** from the same
> folder in client mode. Compute is always server-side; only small artifacts come back.

---

## 0. Prerequisites & glossary

To talk to the server you need four things (all produced at deploy time):
- **`BOX`** — the box's host/IP and port (default `8443`), e.g. `https://172.x.x.x:8443`.
- **Client mTLS material** — `certs/ca.pem`, `certs/client.pem`, `certs/client.key`
  (generated by `bootstrap/gen_certs.py` during `up`; copy them back to your machine).
- **Bearer token** — the value of `CAPM_P5_TOKEN` (set in `.env` at deploy).
- **The `capm-p5/` folder** on your machine (the same folder, used in *client* mode).

Model aliases (from `config/models.yaml`): `qwen14b` (`Qwen/Qwen2.5-14B-Instruct`),
`phi4` (`microsoft/phi-4`), `qwen32b` (`Qwen/Qwen2.5-32B-Instruct`), `qwen7b`
(`Qwen/Qwen2.5-7B-Instruct`).

## 1. Connecting to the server

There is **no SSH** to the box after deploy — all access is the HTTPS API. Every `/v1/*`
call needs **both** the client cert (mTLS) **and** the bearer token.

**Set up your local client env (once):**
```bash
cd capm-p5/
export CAPM_P5_URL="https://BOX:8443"
export CAPM_P5_TOKEN="<the token from .env>"
# certs/ already contains ca.pem, client.pem, client.key copied back from the box
```

**Verify connectivity (liveness — the only unauthenticated route):**
```bash
curl --cacert certs/ca.pem https://BOX:8443/healthz
# → {"status":"ok","uptime_s":...,"resident_models":[...],"probes":[...]}
```

**First authenticated call (proves cert + token both work):**
```bash
curl --cacert certs/ca.pem --cert certs/client.pem --key certs/client.key \
     -H "Authorization: Bearer $CAPM_P5_TOKEN" \
     "$CAPM_P5_URL/v1/gpu"
```
If `/healthz` works but `/v1/gpu` 401/403s → token or client-cert problem (see §8).

**Or use the bundled client** (wraps the curl + cert + token plumbing):
```bash
./run.sh status --remote "$CAPM_P5_URL"     # health + GPU + jobs
```

## 2. Environment setup

### On the box (one-time, during your access window)
Only Docker + an NVIDIA GPU are required on the box; everything else is inside the image.
```bash
# copy capm-p5/ to the box, then:
cd capm-p5/
cp .env.example .env && edit .env       # set CAPM_P5_TOKEN, port, HF_HOME, CUDA_VISIBLE_DEVICES
./run.sh up                              # preflight → build → fetch models → train+freeze probes → serve
```
`preflight` fails fast if CUDA/VRAM/disk/Docker/network are not satisfied (§8). If the box is
**air-gapped**, pre-stage the model cache into `models/` first (see §8, "air-gap").

### On your machine (client side)
You only need either `curl` + the certs, or a Python ≥3.10 venv for the bundled client:
```bash
python -m venv .venv-client && . .venv-client/bin/activate
pip install httpx pyyaml         # client deps only — no torch needed locally
```

## 3. Starting / stopping services

All run **on the box** (they manage the Docker daemon there):

| Command | Effect |
|---------|--------|
| `./run.sh up` | idempotent bootstrap: build (if needed) → fetch models (skips present) → **train+freeze probes (skips present)** → `docker compose up -d` → healthcheck. Re-running = restart without redoing finished steps. |
| `./run.sh probes [--model M]` | (re)train + freeze a probe only |
| `./run.sh down` | stop + remove the container (models/probes/artifacts on the mounts survive) |
| `./run.sh status` | health + `/v1/gpu` + resident models + frozen-probe inventory |
| `./run.sh logs [-f]` | container logs (follow with `-f`) |

**Daemon behavior:** the container runs with `restart: unless-stopped`, so it survives your
disconnect and **comes back automatically after a box reboot** (Docker restarts it; it
reloads sensors and lazy/warm-loads models per policy). To intentionally stop it: `./run.sh
down`. After a reboot you do **not** re-run `up` (no rebuild/redownload) unless the image or
models changed.

**What "up" does NOT do:** it does not run experiments. It leaves you with a healthy server +
frozen probes. Experiments are triggered separately (§4), typically remotely.

## 4. Running experiments

Experiments are **async jobs** (minutes long) that run server-side on the GPU. You trigger
them and poll/pull results. The full matrix:

```bash
# from your machine — runs the whole grid using the FROZEN probes:
./run.sh matrix --remote "$CAPM_P5_URL"
#   for each model in {qwen14b, phi4, qwen32b}:  b2, d2, f3   + c2 once  + b1 transfer (qwen7b↔qwen14b)
#   polls every job to completion, pulls artifacts into ./artifacts/results/
```

A single experiment:
```bash
./run.sh run d2 --model qwen14b --remote "$CAPM_P5_URL"
# → submits POST /v1/experiments/d2, prints job_id, polls, pulls the CSV+manifest locally
```

What each pass produces (Phase-4 schema, into `artifacts/results/...`):
- **b1** — usage-probe transfer + controls + 7B↔14B cross-size (uses the frozen probe/cache).
- **b2** — fabrication-detection AUC by subtlety.
- **d2** — security–utility frontier (+ relay sample).
- **f3** — adaptive-adversary residuals.
- **c2** — NLI/schema faith table (no LLM; runs anytime).

**Where results land:** locally under `artifacts/results/` on your machine (the client pulls
them); they also persist on the box's mounted `artifacts/` volume. Both are the Phase-4 CSV
columns, so the existing `p4/audit/*` and figure tooling work unchanged.

**Recommended order:** the probes are already frozen at bootstrap, so just run the matrix.
For one model at a time on a shared box, prefer per-model job groups (load → run b2/d2/f3 →
evict) — see Q5/Q6 in §9.

## 5. Calling the API endpoints directly (curl)

Reusable header block:
```bash
AUTH=(--cacert certs/ca.pem --cert certs/client.pem --key certs/client.key
      -H "Authorization: Bearer $CAPM_P5_TOKEN" -H "Content-Type: application/json")
U="$CAPM_P5_URL"
```

**Ops / inventory**
```bash
curl "${AUTH[@]}" "$U/v1/version"                       # git sha, probe hashes, sensor versions
curl "${AUTH[@]}" "$U/v1/gpu"                            # reserved/free VRAM, cap, resident models
curl "${AUTH[@]}" "$U/v1/models"                         # resident models + which have frozen probes
curl "${AUTH[@]}" "$U/v1/probes"                         # frozen probes: model, spec, sig-verified, train F1
```

**Model residency (manual HOT/PARKED/COLD control)**
```bash
curl "${AUTH[@]}" -X POST "$U/v1/models/load"  -d '{"model":"qwen14b","dtype":"bf16"}'  # →HOT
curl "${AUTH[@]}" -X POST "$U/v1/models/park"  -d '{"model":"qwen14b"}'                 # HOT→PARKED (free VRAM)
curl "${AUTH[@]}" -X POST "$U/v1/models/evict" -d '{"model":"qwen14b"}'                 # →COLD (free RAM)
# /v1/models reports each model's tier (hot|parked|cold) + VRAM/RAM held.
```

**Structured errors (every error has this shape — branch on `error`):**
```jsonc
// e.g. HTTP 503 when a load can't fit:
{ "error": "insufficient_vram",
  "message": "Cannot load qwen32b: needs 66.0 GB, only 31.2 GB free under the 70 GB cap.",
  "details": { "requested_model": "qwen32b", "dtype": "bf16",
               "required_gb": 66.0, "available_gb": 31.2, "vram_cap_gb": 70,
               "hot_model": "qwen14b", "parked_models": ["phi4"] },
  "suggested_action": "Park the HOT model (POST /v1/models/park {\"model\":\"qwen14b\"}) then retry; the 32B needs the GPU to itself.",
  "retry_after_s": 60 }
```
Codes → HTTP: `insufficient_vram`/`insufficient_ram` → 503; `probe_missing`/`probe_unverified`
→ 409; `model_not_configured` → 400; `unauthorized`/`forbidden` → 401/403;
`model_load_failed`/`job_failed` → 500 (still enveloped). A failed load **never** crashes the
worker — CUDA-OOM is caught, VRAM freed, and returned as `insufficient_vram` with live numbers.

**Scoring primitive (the atomic CAPM op; returns the probe inference too)**
```bash
curl "${AUTH[@]}" -X POST "$U/v1/score" -d '{
  "model":"qwen14b","dtype":"bf16",
  "ctx":"<source document text>","field":"due_date","value":"2025-01-01",
  "w_decl":0.85,"source_class":"AUTHORITATIVE_API","leaked_premise":false }'
# → {"u":0.97,"s":0.81,"faith":1.0,"g":0.81,"w_realized":0.6885,"binding":"support",
#    "probe":{"p_context_driven":0.97,"dim":5120},"manifest":{...signed...}}
```

**Experiments (async → job_id)**
```bash
JID=$(curl "${AUTH[@]}" -X POST "$U/v1/experiments/d2" \
        -d '{"model":"qwen14b","dtype":"bf16","advisories":30,"seed":7}' | jq -r .job_id)
curl "${AUTH[@]}" "$U/v1/jobs/$JID"                      # status: queued|running|done|error (+progress)
curl "${AUTH[@]}" "$U/v1/jobs/$JID/artifacts" -o d2_qwen14b.zip   # pull CSV+manifest
curl "${AUTH[@]}" "$U/v1/jobs"                           # list all jobs
```

All `/v1/*` are mTLS+token gated; every result embeds a **signed manifest**; there is **no
train/fit endpoint** (probes are frozen at bootstrap).

## 6. Monitoring & logs

- **Health/throughput from anywhere:** `./run.sh status --remote $U` or the `/healthz`,
  `/v1/gpu`, `/v1/jobs` endpoints. Watch `/v1/gpu` `reserved_gb` vs `cap_gb`, and job
  `progress`.
- **On the box:** `./run.sh logs -f` (uvicorn + job runner logs), and `nvidia-smi` to see
  real GPU memory/util and **co-tenant** processes (this is a shared box — others' jobs show
  up here too).
- **Job progress:** each experiment logs claims-processed / total; a stuck job (no progress)
  is the signal to check logs and `nvidia-smi` for OOM or contention.
- **What "idle" looks like:** between requests, GPU **util ≈ 0%** and (in our dormant+parking
  default) **VRAM drops to ~2.5 GB** as the model parks to RAM (see Q3 in §9). `nvidia-smi`
  shows the freed VRAM; the parked weights show up in host RAM (`free -g` / `htop`).

## 7. Debugging & recovery

| Situation | First checks | Recovery |
|-----------|--------------|----------|
| **Server won't start** | `./run.sh logs`; is a frozen probe missing? cert/port? | server fail-closes if a configured model has no signed probe → `./run.sh probes`; fix `.env`/certs; re-`up` (idempotent) |
| **`/healthz` ok but `/v1/*` 401/403** | token value; client cert path | export the right `CAPM_P5_TOKEN`; pass `--cert/--key`; check the client cert is from the deployed CA |
| **`insufficient_vram` on load** | the structured error body (required vs available GB) | park/evict the HOT model (`/v1/models/park`); the 32B needs the GPU to itself; retry after `retry_after_s`. This is a clean 503, not a crash |
| **OOM mid-job** | logs at the failing claim | server catches it → frees VRAM → returns/marks `insufficient_vram`; reduce batch (if batching on); ensure no co-tenant spike; retry the job (idempotent key) |
| **Model load fails** | disk full? cache corrupt? | check `models/` free space; re-fetch that model; confirm HF cache path = `HF_HOME` |
| **Job stuck `running`, no progress** | logs; `nvidia-smi` util | likely contention or a wedged forward → `./run.sh down && up`; re-submit job (cache makes finished passes instant) |
| **Container died** | `docker ps -a`; logs | `restart: unless-stopped` usually revives it; else `./run.sh up`. Reload reads model weights from the mount (~tens of s); **frozen probes + B1 cache survive**, so no retrain |
| **After box reboot** | `./run.sh status` | daemon auto-restarts; if not, `./run.sh up` (skips build/download/probes) |
| **Results not appearing locally** | job `status`=done? | pull explicitly: `/v1/jobs/{id}/artifacts`; check client `artifacts/results/` perms |

**Recovery principle:** the two expensive assets — the **~140 GB model cache** and the
**frozen signed probes + B1 feature cache** — live on mounted volumes and survive container
restarts and reboots. Recovery never retrains a probe or re-downloads a model unless that
artifact is actually missing.

## 8. Common failure cases & fixes

| Symptom | Cause | Fix |
|---------|-------|-----|
| `up` hangs/fails at model fetch | **box is air-gapped** (no HF) | pre-stage: download on a connected host, ship the cache into `models/`; bootstrap detects + skips. Decide this **before** the access window |
| `preflight` red on disk | <~200 GB free | free space or point `HF_HOME`/`models/` at a bigger mount |
| `preflight` red on Docker | not in `docker` group / no nvidia runtime | use `./run.sh up --no-docker` (bare-venv fallback) for the one-shot window |
| CUDA OOM loading 32B | another model still HOT (or co-tenant) | park the HOT model first; 32B (~66 GB) must own the GPU (the sanctioned exception) |
| Two LLMs HOT requested | ≥62 GB and against policy | not allowed — `one_hot_model: true`; the manager parks the current model before promoting the next |
| Slow reactivation | model was COLD (disk), not PARKED | expected on first load; thereafter it parks to RAM and reactivates in ~3–6 s (§9 Q3) |
| Probe signature verify fails | tampered/incomplete `probe_*.pkl` | re-run `./run.sh probes --model M`; server stays fail-closed until valid |
| 403 from a fresh client | client cert not from the deployed CA | re-copy `certs/client.*` from the box; certs are per-deployment |
| Port unreachable from your machine | firewall / wrong interface | confirm inbound `8443` during the access window; bind interface in `server.yaml` |
| HF download throttled | rate limits on large pulls | resume (`up` is idempotent); or pre-stage |
| Slow first request after idle | model was COLD (disk) | only the *first* load is slow; once parked it reactivates from RAM in ~3–6 s. Pre-load before a batch with `/v1/models/load` if you want a hot start |

## 9. Resource model & lifecycle (answered for *our* setup)

These answers are specific to our box (**128 cores · 256+ GB RAM · 96 GB VRAM**, shared), our
models (Qwen2.5-14B ≈ 29.5 GB, Phi-4-14B ≈ 29 GB, Qwen2.5-32B ≈ 64 GB, Qwen2.5-7B ≈ 15 GB, all
bf16), and our sensors (all-MiniLM ≈ 0.1 GB, DeBERTa-v3 base+large ≈ 2.3 GB; frozen probes are
KB).

### Q1 — How much VRAM, and **why is it so large** (is it weights, probe, KV, overhead)?
**It's almost entirely model weights — and that's fundamental to bf16, not overhead.** bf16 =
**2 bytes per parameter**, so the weights alone are:
| Component | 14B | 32B | note |
|-----------|-----|-----|------|
| **Weights (bf16, 2 B/param)** | **~29–29.5 GB** | **~64–65 GB** | the dominant term — ~97% of the total |
| Frozen probe | ~20 KB | ~20 KB | a logistic-regression vector — negligible |
| Sensors (shared, loaded once) | ~2.5 GB | ~2.5 GB | MiniLM + DeBERTa base+large, not per-model |
| KV cache (our ~500-tok prompts, ~36-tok gen) | ~0.1 GB | ~0.15 GB | tiny — we don't do long generation |
| Activations (`output_hidden_states` forward) | ~0.25 GB | ~0.35 GB | transient, per sequence |
| CUDA context + PyTorch reserved/fragmentation | ~1–3 GB | ~1–3 GB | framework overhead |
| **Total resident** | **~33–35 GB** | **~66–68 GB** | |

So **32B ≈ 66 GB because 32.5B params × 2 bytes ≈ 65 GB of weights** — the probe, KV cache, and
activations together add <4 GB. Nothing is bloated. The only lever to shrink it is
*quantization* (8-bit → ~32 GB, 4-bit → ~18 GB), which we **deliberately rejected** for the
32B so its probe reads true bf16 hidden states and the 14B→32B scale read isn't confounded by
precision. (If VRAM were ever the blocker, 8-bit 32B is the fallback — at the cost of the
probe-on-quantized caveat.)

Per-configuration totals (one HOT model at a time):
| Serving config | VRAM | Used? |
|----------------|------|-------|
| 1× 14B + sensors | **~33–35 GB** | yes (routine) |
| 1× 32B + sensors | **~66–68 GB** | yes (the sanctioned exception) |
| any two LLMs HOT together | ≥ ~62 GB | **no — one HOT at a time** |

**Adopted policy:** **one model at a time, dormant by default.** Routine 14B work stays
**≤ ~50 GB** (the soft ceiling); **Qwen2.5-32B bf16 (~66 GB) is the single sanctioned
exception**, run alone and sequentially. We never co-reside two LLMs. Config: `soft_ceiling_gb:
50`, hard `vram_cap_gb: 70` (admits the 32B), `resident_pool_max: 0`, `one_model_at_a_time:
true`. Bootstrap probe-training also loads one model at a time (peak = the 32B at ~66 GB). You
never need the full 96 GB — most of the time you use ~33–35 GB (a 14B) or ~2.5 GB (idle).

**Cooldown / longer idle window (your Q2 — yes, and we do better than a long VRAM hold).**
With **256 GB RAM** we add a **PARKED** tier: on idle a model moves VRAM→CPU RAM (`.to('cpu')`)
— this **frees the VRAM in seconds** (good shared-box citizen) yet **reactivates in ~3–6 s**
(`.to('cuda')`), versus ~30–120 s cold from disk. RAM holds **all** our models parked at once
(~123 GB ≪ 256). So instead of keeping a model HOT for 20+ minutes (hoarding 33–66 GB of
VRAM), we **park** it and keep it parked for a long window — cheap, and reactivation is nearly
instant. Knobs: `idle_park_seconds: 120` (HOT→PARKED), `idle_evict_seconds: 1800` (PARKED→COLD,
or `null` to keep parked indefinitely). If you specifically want a model VRAM-HOT longer, set
`resident_pool_max: 1` + a large `idle_evict_seconds` — fine for a single 14B on the 96 GB box
— but **parking is the better default** here.

### Q2 — Will the models run *continuously* once deployed?
The **server process** runs continuously (Docker daemon, `restart: unless-stopped`). The
**models do not *compute* continuously** — a loaded model just sits in VRAM and runs a
forward pass only when a request arrives. So: continuous *service*, on-demand *computation*.
Whether the *weights stay in VRAM* between requests is a policy choice (Q3/Q5).

### Q3 — Do they occupy GPU/system resources when idle?
**In our adopted (dormant + parking) policy: no VRAM is held idle.** When a model goes idle it
**parks to CPU RAM**, so:
- **VRAM (idle):** ≈ **2.5 GB** (just the resident sensors), or ≈ **0** with `sensor_device:
  cpu` — i.e. ~93 GB handed back to co-tenants between jobs.
- **RAM (idle):** the parked weights live here (~29–64 GB per parked model; 256 GB has room
  for all). This is cheap RAM we have in abundance, traded for the scarce VRAM.
- **GPU compute:** ≈ **0%** between requests.
- **CPU:** negligible when idle (async server); the 128 cores are used only during sensor/data
  work.

(If you instead chose the **always-resident** policy, a HOT 32B *would* hold ~66 GB of VRAM
continuously even while doing nothing — which is exactly why we default to dormant+parking on
a shared box.)

### Q4 — Estimated setup time and runtime latency?
**One-time setup (on the box, during the access window):**
- Docker image build: ~5–15 min.
- Model download (~140 GB): network-bound — ~20–60 min at typical speeds; **~0 if pre-staged**
  (air-gapped path).
- **Probe training (load each model + extract B1 features over 80 advisories + fit + freeze):**
  ~3–10 min/model → **~15–40 min** for the four. (One-time; frozen forever after.)
- Server start (sensors + first model): ~1–2 min.
- **Total: ~45–90 min realistic** (download + probe-train dominate), much less if pre-staged.

**Runtime latency (steady state):**
- `/v1/score`, **HOT** model: ~**0.1–0.5 s** (one forward + frozen LR <1 ms + embed ~ms + NLI ~10–50 ms).
- `/v1/score`, **PARKED** model (reactivate from RAM): + ~**3–6 s** (`.to('cuda')`).
- `/v1/score`, **COLD** model (first load from disk): + 14B ~**30–60 s**, 32B ~**60–120 s**.
- An **experiment job** (B2/D2/F3): hundreds of forward passes → **minutes** per pass at 14B,
  more at 32B (current one-at-a-time loop; the optional batching milestone cuts this 5–20×).
  A cold model's load cost is amortized to near-nothing over a multi-minute job.

### Q5 — Can we keep models *dormant* and load on demand per request?
**Yes — and it is the ADOPTED DEFAULT**, not just an option. `resident_pool_max: 0` +
`idle_evict_seconds: 600`. The flow: server stays live holding only sensors (~2.5 GB) + the
KB-sized frozen probes → a request arrives → the model is **lazy-loaded** (one at a time) →
forward/pipeline runs → result returned → after the idle TTL the model is **unloaded and the
GPU returns to dormant**. Only the big **LLM weights** are dormant; **probes + sensors stay
resident**. Because our real workload is **minutes-long batch experiment jobs** — and each job
runs as a subprocess that loads its model once and frees it on exit — the ~30–120 s load is
amortized to near-zero per job. Dormant is a genuinely good fit here, not a compromise.

### Q6 — Dormant vs always-loaded: the tradeoffs for *us* (and why we chose dormant)
| | **Always-resident (warm)** | **Dormant (adopted)** |
|---|---|---|
| Idle VRAM | holds 33–66 GB continuously | **~2.5 GB** (sensors only); ~93 GB back to co-tenants |
| First-request latency | ~0.1–0.5 s | + 30–60 s (14B) / 60–120 s (32B) cold load |
| Batch-job latency | load paid once | load paid once per model per job (amortized over minutes) |
| Shared-box citizenship | poor (hoards VRAM idle) | **good** (occupies only while working) |
| Thrash risk | none | real if you alternate models rapidly → mitigated by the TTL + per-model job grouping |
| Best for | frequent interactive `/score` | **our matrix** (long, sequential, per-model passes) |

**Adopted configuration:** **fully dormant** — sensors + frozen probes resident (~2.5 GB),
**no LLM at rest**, lazy-load **one model at a time** with a 10-min keep-alive
(`resident_pool_max: 0`, `idle_evict_seconds: 600`, `one_model_at_a_time: true`,
`soft_ceiling_gb: 50`, `vram_cap_gb: 70`). Drive the matrix **one model at a time** (qwen14b →
b2/d2/f3 → evict → phi4 → … → qwen32b). Each model's load is paid **once per job**, peak VRAM
is a single model (~33–35 GB for 14B; the **one sanctioned ~66 GB exception** for 32B), and
the box is idle-frugal — which is the whole point on a shared host. Bump `resident_pool_max`
to 1 only if you start making many *interactive* `/score` calls against one model. These knobs
live in `config/server.yaml`.
